## 一、技术栈

| 层级 | 技术 |
|------|------|
| Web 框架 | FastAPI 0.1.0 |
| RAG 编排 | LangChain（chain.py 使用 LCEL 管道语法） |
| 向量数据库 | Qdrant（通过 `langchain-qdrant` 封装） |
| Embedding 模型 | `text-embedding-bge-small-zh-v1.5`（OpenAI 兼容 API） |
| LLM | 支持双模式：本地 LM Studio / Ollama 或云端 API（如 DeepSeek） |
| 文档解析 | PyPDF2（PDF） + TextLoader（txt/md） |
| 文本切分 | `RecursiveCharacterTextSplitter`（chunk_size=800, overlap=120） |
| 配置管理 | `pydantic-settings` + `.env` 文件 |
| 日志 | Python `logging` 模块，统一格式，含每步耗时统计 |
| 异步 | FastAPI `async` endpoint（文件上传） |

## 五、API 接口概览

| 方法 | 路径 | 描述 | 文件 |
|------|------|------|------|
| `GET` | `/` | 根路径，返回运行状态 | `app/main.py` |
| `GET` | `/health` | 健康检查，返回 `{"status":"ok"}` | `app/api/health.py` |
| `POST` | `/documents/upload` | 上传文档（PDF/txt/md），入库 | `app/api/documents.py` |
| `POST` | `/rag/query` | RAG 问答，返回答案+来源 | `app/api/rag.py` |

### POST /rag/query

**Request:**
```json
{
  "question": "什么是 RAG?",
  "top_k": 5
}
```

**Response:**
```json
{
  "question": "什么是 RAG?",
  "answer": "RAG 是检索增强生成...",
  "sources": [
    {
      "content": "...",
      "source": "intro.pdf",
      "page": 3,
      "score": 0.87
    }
  ]
}
```

### POST /documents/upload

**Request:** `multipart/form-data` (file field)

**Response:**
```json
{
  "filename": "intro.pdf",
  "file_path": "data/raw/a1b2c3d4.pdf",
  "pages": 12,
  "chunks": 45,
  "status": "indexed"
}
```

## 七、分层架构原则

```text
┌─────────────────┐
│   api/          │  ← 路由层：只处理 HTTP 请求/响应，不写业务逻辑
├─────────────────┤
│   services/     │  ← 业务层：编排调用，连接 API 和 RAG 核心
├─────────────────┤
│   rag/          │  ← RAG 核心层：loader, splitter, embeddings, retriever, prompt, chain
├─────────────────┤
│   llm/          │  ← 模型调用层：LLM 实例创建（本地/云端切换）
├─────────────────┤
│   core/         │  ← 配置层：env 变量、日志
├─────────────────┤
│   schemas/      │  ← 数据结构：Pydantic 请求/响应模型
├─────────────────┤
│   utils/        │  ← 工具函数：文件处理、ID 生成
└─────────────────┘
```

**调用方向：** `api → services → rag/llm/core`，上层依赖下层，下层不感知上层。

## 九、设计亮点

1. **LLM 双模式切换** — 通过 `llm_provider` 配置项一键切换本地/云端，无需改业务代码
2. **全流程耗时统计** — 6 个步骤均有 `[RAG][STEP X]` 开始/完成日志，精确到毫秒
3. **LCEL 管道语法** — `rag_prompt | llm | StrOutputParser()` 清晰表达 RAG 链
4. **文件安全** — 上传文件名使用 UUID 重命名，扩展名白名单校验
5. **清晰分层** — API → Service → RAG Core，职责分明，便于扩展
